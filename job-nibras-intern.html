The internship

A trained model that cannot be deployed is worth nothing. NIBRAS is the layer that takes what the model teams build and makes it run in real environments: factory floors, hospital corridors, government offices, offline server rooms. The work is packaging, optimization, and delivery.

You will take trained models from FAHM, SIMAH, and SIJILL and make them production-ready. That means converting them to portable formats, compressing them for target hardware, wrapping them in inference APIs, and verifying they perform correctly under real conditions. The work is infrastructure. It is precise, measurable, and consequential.

Three months. On-site in Lahore. You work directly alongside the founding team. No support queue, no separate room. The work happens in the same space, together.

Model format
ONNX Runtime
The portable inference format. You will export trained PyTorch models to ONNX and verify they produce identical outputs on different hardware.
Optimization
HuggingFace Optimum
Quantization and graph optimization. INT8 and INT4 compression. You will measure accuracy and latency before and after every change.
Containerization
Docker + FastAPI
Each model runs inside a container behind a FastAPI inference server. You will write the Dockerfile, the API, and the health checks.
Delivery pipeline
GitHub Actions + Linux
CI/CD for models. Automated testing on push, artifact versioning, deployment to target environments. The pipeline that makes delivery repeatable.
What you will do
  • 01 Export trained models to ONNX and verify output correctness. Take a model trained in PyTorch on a GPU cluster and make it run portably. You verify that the exported model produces numerically identical outputs to the original. Differences are bugs, not estimates.
  • 02 Apply quantization and measure the accuracy-speed tradeoff. Compress model weights from FP32 to INT8 or INT4. Benchmark latency and memory before and after. Document exactly what was lost in accuracy and what was gained in speed. The numbers are the decision.
  • 03 Build Docker containers and FastAPI inference servers for each model. Write the Dockerfile. Wrap the model in a FastAPI endpoint with proper input validation, error handling, and structured JSON responses. Test that the container runs identically on different machines and operating systems.
  • 04 Build and maintain the CI/CD pipeline for model deployment. Automated tests on every model update: output correctness, latency within threshold, container build success. A model that breaks silently in production is worse than a model that was never deployed. The pipeline catches it first.
  • 05 Benchmark models on target hardware and write deployment runbooks. How fast does this model run on the actual server where it will live? How much RAM does it use at peak load? You will run these numbers on real hardware and write the documentation that tells the next person exactly how to deploy it.
Who this is for
  • You write Python that other people can run. Scripts that work outside notebooks, handle bad input gracefully, and can be read by someone who did not write them. Infrastructure code, not research code.
  • You understand what a machine learning model actually does. You do not need to have trained one. You need to understand what inputs it takes, what outputs it produces, and why those outputs might be wrong. One project where you used or integrated a model is enough.
  • You are comfortable on Linux. Files, processes, environment variables, SSH, bash scripts. You can read an error message, find the cause, and fix it without a GUI.
  • You have used Docker or are ready to learn it fast. Containers, images, ports, volumes. If you have not used Docker before, say so, and tell us what you have used for deployment or environment management instead.
  • You are methodical under uncertainty. Deployment work has many failure modes and not all of them are obvious. You run tests, read logs, form a hypothesis, and verify it. You do not guess and move on.
Strong interns are offered a full-time role when the internship ends. The path is direct and based entirely on what was built.
Terms

Three months. On-site in Lahore. The internship is paid. The amount is discussed during the selection process and reflects the work involved.

You will work alongside the founding team every day on systems that are actively being built and deployed. There is no support queue, no task-assignment portal. The work happens in the room, together, and you are expected to participate fully, not observe.

Apply

Send your CV and a description of one system you built and deployed. Not a model you trained. Something you made run reliably in an environment that was not your own laptop. What was it, how did you deploy it, what broke, and what did you fix.

If you have experience with Docker, ONNX, quantization, or inference APIs, describe it specifically. Vague references to having used these tools are not useful. Tell us what you built with them.
